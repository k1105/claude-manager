# システムv2.0 — 「記録駆動型マネージャー」への転換

_このお便りは、claude-managerを既に運用中の方に向けたアップデートガイドです。_
_直接git pullするのではなく、あなたのClaudeにこのファイルを読ませて、環境に合わせた適用を相談してください。_

---

## 何が変わったか（一言で）

**「監視する仕組み」から「記録し、構造化し、穴を見つけて駆動する仕組み」への転換。**

## なぜ変えたか

2週間の運用で見えた問題：
1. セッションを跨ぐと会話の中で決まったことが消える
2. プロジェクトの状態を横断的に把握する仕組みがない
3. screen-log-checkが1日90回のcueを消費するが、離脱アラートはほぼ発火しない
4. マネージャーが受動的（聞かれたら答える）で、自分から穴を突けない
5. リサーチとプロジェクトが分離していて情報が散在する

## 二大方針（最も重要）

**① ユーザーの脳を駆動させ続ける。** プロジェクトの「穴」を見つけてそこを突く。ユーザーが自分から動くよりも、マネージャーに駆動されて動く仕組みを作る。

**② ユーザーの手を離れて進められることを最大化する。** 調べたらわかること、整理したらできることは自律的に実行する。論点整理・現状可視化・情報収集はマネージャーの仕事。アイデアを考えるのはユーザーの仕事。

## 主な変更点

### 1. status.md の導入（最大の変更）

全プロジェクトに `status.md` を追加。これが日常的に最も読み書きするファイルになる。

```markdown
# {プロジェクト名} Status

## 現在地
- Phase: 企画中 / 制作中 / 待ち / 完了 / 保留
- Next milestone: 日付＋内容
- Blocker: あれば
- Last updated: YYYY-MM-DD

## ゴールとのギャップ
- ゴール: 次の到達点
- 現状との差: 何が足りないか
- → 穴: 最も重要な未解決ポイント

## Claudeが自律的にできること
- [ ] リサーチ、情報収集、整理等

## ユーザーと一緒にやること
- [ ] 意思決定、壁打ち、制作等

## 決定事項ログ
- YYYY-MM-DD: 何が決まったか
```

**ポイント：**
- 「ゴールとのギャップ」でプロジェクトの穴が構造的に可視化される
- 「Claudeが自律的にできること」で、マネージャーが勝手に進められるタスクが明示される
- briefing/checkinではstatus.mdを読めば全体が分かる

### 2. deadlines.md の導入

`projects/deadlines.md` に全プロジェクト横断のデッドラインを一元管理。固定日付と周期型の2部構成。

**「未定」は許容しない。** 新しい「やりたい」が出たら「暫定的であっても、いつ？」と聞く。

### 3. research/ 廃止 → projects/ 統合

リサーチは全てプロジェクト配下で管理。
- プロジェクト固有 → `projects/{name}/research/`
- 複数プロジェクト横断 → `projects/_shared_research/`

理由：リサーチは必ず何かのプロジェクトに与するもの。分離していると散在して見落とす。

### 4. plans/ 廃止

- goals.md → projects/*/status.md で代替（横断俯瞰は都度生成）
- weekly review → logs/weekly/ に移動（記録であり計画ではない）
- principles → docs/principles.md に移動

### 5. logs/ 整理

```
logs/
  daily/YYYY-MM-DD.md    # 日次ログ
  weekly/                 # 週次レビュー
  meetings/               # 会議記録
  screen/                 # スクリーンキャプチャ
  system/                 # システムログ（*.log）
```

### 6. screen-log-check cue 廃止

10分おきのcueを廃止（-90cue/日）。離脱判定はcheckin（2-3時間おき）に統合。screen logの日次統計はend-of-dayで集計。

### 7. 新スキル

- **project-scan**（週1回）— 全status.md横断スキャン。デッドライン接近、放置検出、自律タスク検出、criteria未設定チェック
- **autonomous-prep**（デッドライン駆動）— 会議や提出物に向けて論点整理・現状可視化・情報収集を自律実行

### 8. realtime-record.md ルール

会話のまとまりごとに決定事項をファイルに即時記録する。「あとでまとめて書く」は禁止。

---

## 適用方法

1. このファイルをあなたのClaudeに読ませる
2. Claudeと一緒に、自分の環境に合わせてどこから適用するか決める
3. 最も効果が高いのは **status.md の導入**。まずアクティブなプロジェクト2-3個に入れるだけで変わる
4. ファイル構造の変更（research/廃止等）は大きいので、状況に合わせて段階的に

**全部一度にやる必要はない。** 一番効くのはstatus.md。次にdeadlines.md。構造変更は最後でいい。
